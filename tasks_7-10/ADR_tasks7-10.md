### Решения

#### 1. Общие решения
- из-за прогнозируемого роста кол-ва данных, паттернов использования, а также возможности горизонтального масштабирования решено сразу предусмотреть шардирование БД MongoDB;
- требуется репликация MongoDb для обеспечения надежности и возможности реализации паттерна read replica;
- предпочтительно хэшированные ключи шардирования;
- понадобятся вторичные индексы для ускорения отдельных запросов;
- требуется кэширование, в первую очередь данных карточек товаров для просмотра;
- возможно использование CDN для ускорения доступа пользователей к сайту;

#### 2. Общие альтернативы
- добавлять кэширование и CDN не сразу;
- range стратегии шардирования для отдельных коллекций могут, например, фильтровать продукты по артикулам; но потребуется осторожная настройка балансировки;

#### 3. Схемы коллекций
##### orders
```mongodb-json
{
  _id: UUID,
  user_id: NumberLong, 
  created_at : Date,
  items : [{
    product_id: "uniq_product_id",
    order_in_chart: "NumberInt",
    "price": "Double"
  }],
  total_sum: NumberDecimal,
  geo_zone: String
}
```
Ключ шардирования - `{ user_id: 'hashed' }` + выключенная балансировка.
Стратегия шардирования: хэшированное шардирование.
**Обоснование** 
  - поиск заказов по пользователю, хорошее распределение по чанкам и шардам (число заказов у похожих по активности пользователй в среднем одинаково), все заказы всегда на одном шарде.
    - недорогой поиск отдельного заказа для обновления статуса после фильтрации всех заказов по пользователю: используя индекс по primary_key (_id); 
- **Альтернативы**:
  - ключ шардирования `{ user_id: 'hashed', created_at: Date }` - более равномерное распределение по чанкам.
  - ключ шардирования `{ geo_zone: 1, user_id: 'hashed', created_at: Date }` - заказы будут привязываться к зоне + юзеру; плюс - можно реализовать zone-sharding для orders;

##### products
```mongodb-json
{
  _id: String,
  category_id: NumberInt,
  name: String,
  description: String,
  price: NumberDecimal,
  stocks: [{
    geo_zone: String,
    quantity: NumberInt,
  }],
  params : {
    color : String,
    size : NumberInt,
    ....
  }
}
```

Ключ шардирования - `{ category_id: hashed, _id: 1 }` + выключенная балансировка.
Стратегия шардирования: хэшированное шардирование.
- **Обоснование**: 
  - много запросов поиска по категориям, большие категории делятся на чанки, но без балансировки 1 категория на одном шарде;
  - в случае отдельных больших категорий возможна либо ручная балансировка, либо включение автоматической балансировки позднее;
  - запросы изменения кол-ва остатков также будут возможны с таким ключем;
  - фильтрация по цене - с помощью вторичного индекса по цене;
- **Альтернативы**:
  - `{ category_id: 'hashed' }` - без балансировки все товары в одной категории только на 1 шарде, минусы - рост размера без возможности параллельной обработки;
  - `{ category_id: 'hashed', price: 1 }` - поиск по категории и цене без вторичных индексов, минусы - возможно неравномерное разделение на чанки по цене, поскольку только первая часть составного клбюча может хэшироваться;


**carts**
```mongodb-json
{
  _id: UUID,
  owner_id: String, //session_id(UUID) или user_id (NumberLong) для залогиненых пользователей
  items: [{
    product_id: UUID,
    price: NumberDecimal
  }],
  status: String,
  created_at: Date,
  updated_at: Date,
  expires_at: Date
}
```
Ключ шардирования - `{ owner_id: hashed}` + выключенная балансировка.
Стратегия шардирования: хэшированное шардирование.
- **Обоснование**: 
  - получение текущей корзины по user_id/session_id, а также равномерное распределение данных по чанкам/шардам.
- **Альтернативы**:
  - добавить geo_zone и ключ шардирования  `{ geo_zone: 1, owner_id: hashed}`

## Задание 8. ADR. Выявление и устранение «горячих» шардов
> Из-за категории «Электроника» произошла перегрузка одного из шардов MongoDB, так как 70% запросов приходилось именно на эти товары. 
> Поэтому сейчас нужно разработать стратегию, как выявлять и устранять такие «горячие» шарды, а ещё предложить метрики мониторинга, чтобы в будущем можно было предотвращать такие ситуации. 
> Не забудьте учесть, что товары из популярных категорий могут создавать непропорциональную нагрузку на отдельные узлы.
 
### Как выявить горячие шарды?
Пока не включена балансировка, необходимо вынести в метрики по каждой шардированной и реплицированной ноде монго:
- `db.serverStatus().opcounters` - кол-во различных операций per shard;
- `sh.status({verbose: 1})` - кол-во чанков;
- стандартные метрики CPU/memory/IO;
- метрики response time запросов;
К этим метрикам следует настроить простые алармы на превышение уставок, либо, например, хронологические алармы на динамику роста значений на разных шардах;

### Как устранить?
1. Вероятно, необходимо разделить данные по разным шардам. 
   - Например, из-за большой читающей нагрузки к коллекции products, где при использовании составного ключа `{ category_id: hashed, _id: 1 }` для отдельных категорий может накопиться слишком много чанков. 
   - Часть этих чанков вручную или автоматически следует перенести на другой шард. 
   - Судя по документации монго будет стараться минимизировать кол-во шардов, на которых будет размещены данные. 
2. Возможно, не был реализован паттерн `read replica`, тогда следует перенаправить читающие запросы (скажем к `products`) на реплики.
3. Горизонтальное масштабирование: добавить новый шард.

## Задание 10. ADR. Миграция на Cassandra: модель данных, стратегии репликации и шардирования
> Во время «чёрной пятницы» интернет-магазин использовал MongoDB с шардированием на основе диапазонов (Range-Based Sharding). 
> При резком увеличении нагрузки (50 000 запросов/сек.) возникла высокая задержка при масштабировании:
>
> При добавлении новых шардов MongoDB полностью перераспределяла данные между всеми узлами, что вызывало просадку latency в пик нагрузки, 
> так как система тратила ресурсы на перемещение данных.
>
> Руководство решило перейти на БД Cassandra, чтобы обеспечить:
> - Высокую отказоустойчивость (leaderless‑репликация).
> - Быстрое горизонтальное масштабирование без полного перераспределения данных.
> - Равномерное распределение данных

### Задание 10.1 - Анализ
> Определите, какие части данных интернет-магазина (например, заказы, товары, корзины, история заказов, пользовательские сессии) являются критически 
> важными с точки зрения целостности и скорости обработки.
> Обоснуйте, для каких именно данных применение Cassandra имеет смысл (учтите требования к масштабируемости, геораспределённости, скорости записи/чтения и целостности данных).

### Решение
#### Обоснование перехода
Рассмотрим возможные преимущества перехода на Cassandra в контексте задачи.
1. Лучшая отказоустойчивость:
- Cassandra поддерживает leaderless replication, т.е. любой узел может принимать запись.
- MongoDB всё же требует primary в каждом реплицированном наборе. При его падении — возможен failover (с задержкой).

2. Лучшая реализация геораспределённости, что может сократить latency:
- Cassandra может располагать реплики в разных регионах и обрабатывать запросы на ближайшей ноде, без строгого мастера.
- MongoDB, вероятно, это делает хуже (до версии 7.0 не было глобальных write zones).

3. Требуется минимальная latency и максимальный throughput на запись:
- Cassandra позволяет обрабатывать сильно больше событий в секунду (в сравнении с Mongo), потому что запись лог-ориентированная и не требует чтения (write-only workloads, как carts, user_sessions, logs).

#### Что переносить?
| **Коллекция**  | **Перенос в Cassandra**    | **Обоснование**                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
|:--------------:|:---------------------------|:-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|     orders     | Возможен                   | - важны актуальные данные при чтении<br/>- средний RPS на запись матчится с возможностями Cassandra, <br/>- отсутствие failover - пропадут ошибки при записи или чтении, например, нестабильные чтения stale reads<br/>- отсутствие failover - не будет увеличения latency во время повышения нагрузки на новый primary<br/>- снижение latency из-за лучших возможностей для реализации шардирования и роутинга запросов по геозонам<br/>- **появятся новые сложности (см. ниже)** |

**Итого**: предположим, принято решение перенести `carts`, `orders`, `orders_history`, `users_sessions`.

**Важное примечание**:
- важно учесть новые сложности при переносе `orders` в Cassandra; 
- нужно по-прежнему обеспечивать "транзакционность" между `orders` и `products`: в момент создания заказа `orders` остатки товаров должны быть проверены и уменьшены в `products`; если остатков не хватает, или возникла ошибка при создании заказа, транзакция должна быть откачена;
- в mongo это было возможно сделать в рамках единой транзации, в Cassandra даже при переносе обеих коллекций транзакцию сделать не получится.
- вариантов 2: 
  - переносить в Cassandra только `carts` и `user_sessions`, но тогда не очень понятно, зачем вообще использовать новую БД;
  - реализовать распределенную транзакцию между `orders` и `products` в разных БД; для этого может пригодиться:
    - изменение флоу: резервирование остатка -> оформление заказа -> снижение остатков (или разморозка резерва при ошибке создания заказа);
    - новые статусы для заказа (CREATED, CONFIRMED), очередь для event-driven (минусы - dlq и увеличение latency для пользователя) или сервис координатор, который должен будет уметь проводить компенсирующие транзакции, если что-то пошло не так. 

### Альтернативы
На самом деле переход на Cassandra неоднозначен, потому что:
1. Проблема перемещения данных может быть решена:
   - хэш-ключами шардирования вместо ключей с range стратегией; для этого существующую коллекцию можно пересоздать с новым ключом.
   - ручной настройкой балансировки; выглядит, автоматическая балансировка во время черной пятницы и одновременного добавление шарда - не очень верное решение.
2. Кажется, что RPS 50к не должен стать предельным значением для правильно настроенной Монго с шардированием и репликацией. Необходимо проверить и полечить такие проблемы:
   - горячие шарды; лечение - перебалансировка данных;
   - основной объем из 50к RPS - это чтение `products`; тут нужно рассмотреть:

**Активные заказы**
- создание, изменение, удаление. 
- ключ партицирования с учетом геозоны; далее поиск по order_id для изменения конкретного заказа.
```cassandraql
CREATE TABLE orders (
        geo_zone TEXT,
        user_id BIGINT,
        order_id UUID,
        items LIST<FROZEN<order_item>>,
        total_sum DECIMAL,
        status TEXT,
        created_at TIMESTAMP,
        updated_at TIMESTAMP,
        PRIMARY KEY ((geo_zone, user_id), order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);

CREATE TYPE order_item (
       product_id TEXT,
       order_in_cart INT,
       price DOUBLE
     );
```
Завершенные заказы 
- создание, реже изменения (например, возвраты). 
- ключ партицирования с учетом геозоны; 
  - далее поиск по дате (вывести заказы на период);
  - затем по order_id (отобразить детали конкретного заказа).

```cassandraql
CREATE TABLE orders_history (
    geo_zone TEXT,
    user_id BIGINT,
    created_at TIMESTAMP,
    order_id UUID,
    items LIST<FROZEN<order_item>>,
    total_sum DECIMAL,
    status TEXT,
    PRIMARY KEY ((geo_zone, user_id), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id DESC);
```
